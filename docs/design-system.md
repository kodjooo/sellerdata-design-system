# Дизайн-система Sellerdata — индекс для агента

> ⭐ **Для генерации НОВОГО UI (страницы/попапы/элементы) — главный документ:**
> [`docs/ds-ai-style-guide.md`](ds-ai-style-guide.md) — стиль-язык, выверенный по реальному коду
> (токены + правила + эталонные паттерны). Гибрид: генерируй на токенах/паттернах; для сложного
> (таблица, модалка) — бери готовые `Components/Ds/DsTable.vue`, `DsModal.vue`.

Этот файл — **навигатор**, а не справочник. Он не дублирует значения токенов: детали
всегда читаются из первоисточника. Цель — чтобы агент перед вёрсткой открыл нужные
файлы и взял из них точные имена токенов и правила применения, а не «общую идею».

## Правило вёрстки (обязательно)

Перед любой правкой UI в `.vue`/`.scss` агент **обязан прочитать первоисточник целиком**:

1. `resources/js/Pages/DesignSystem/Shared/tokens.js` — полный реестр токенов.
   У каждого токена есть поле `note` — это **правило, когда его применять**. Берём токен
   по смыслу из `note`, а не «похожий по цвету».
2. `resources/scss/_variables.scss` (блок `:root`) — канонический источник значений
   (Tier 1 primitives → Tier 2 semantic). **Это single source of truth для значений токенов.**

## Приоритет источников (ВАЖНО)

Если верстаешь конкретный **компонент** (кнопка, инпут, бейдж, модалка, карточка) —
**эталонный компонент дизайн-системы важнее заметок в `tokens.js`**. Порядок такой:

1. **Эталонный компонент** (`Components/Buttons.vue` и т.п.) — какие именно токены он
   применяет к этому элементу (радиус, размер, шрифт, отступы). Это истина для компонента.
2. `tokens.js` `note` — для выбора токена по смыслу, когда эталона нет.
3. `_variables.scss` — точные значения.

Заметки в `tokens.js` описывают токен «в среднем» и **могут противоречить** конкретному
компоненту. Всегда сверяйся с компонентом/спецификацией, а не только с `note`.
Пример (исправлено): кнопки используют `--radius-sm` (4px), а `--radius-md` (8px) — для
карточек/модалок. Это уже зафиксировано в `note` обоих токенов.

Полные матрицы компонентов (`variant × size × state` на токенах) — в
**`docs/design-system-canon.md`** (Фаза 3). Это рабочая спецификация: при вёрстке нового
компонента бери параметры оттуда.

## Дизайн-система ≠ текущий дашборд (legacy)

Большая часть существующего UI (дашборд, склад) свёрстана на **legacy-классах**
(`.btn-default`, `.btn-regular`, `$blue`, `$peach-*` …) — это «как было». Дизайн-система —
это «как должно стать». Поэтому новая вёрстка по системе может **намеренно отличаться**
от текущего дашборда (например, primary-кнопка по системе имеет тень `--shadow-button`,
а legacy-кнопка — без тени). Это не ошибка. Значения токенов при этом совпадают с legacy
(`$blue` == `--blue-600`), расходится только применение. Не «подгоняй» новую вёрстку под
legacy-дашборд и не правь legacy под систему без отдельной задачи на миграцию.

Запрещено:
- сырые hex-цвета в коде (`#4154F1`) — только `var(--brand)` и т.п.;
- «магические» px для отступов/размеров — только шкала `var(--size-*)` (всё кратно 2px);
- произвольные радиусы — только `--radius-sm|md|lg|full` (кнопки/теги = `sm`, карточки/модалки = `md`);
- сырые `z-index`, длительности transition, opacity — только токены (`--z-*`, `--transition-*`, `--opacity-*`);
- ввод нового цвета/отступа/радиуса. Нет подходящего токена → **сначала** добавить его
  в `_variables.scss` (`:root`) И в `tokens.js`, потом использовать. Не плодить дубликаты.

## Additive-режим: новое строим, legacy НЕ трогаем

Текущий этап — **только additive**. Новые элементы/страницы верстаем на токенах и в новых
файлах. **Существующий legacy-код (старые классы `.btn-*`, `$`-переменные, старые SCSS-
партиалы) НЕ переписываем** — миграция legacy вынесена в отдельный будущий этап.

- Новый компонент → новый `.vue` (scoped-стили) или новый SCSS-партиал.
- Токены — только дополнять в `_variables.scss`/`tokens.js` (это безопасно, additive).
- Не смешивай в одном новом блоке старые legacy-классы и новые токены.

## Типографика — 9 готовых классов

Не задавать `font-size`/`font-weight`/`line-height` вручную — вешать класс из `_type.scss`:

| Класс | Размер / вес | Когда |
|---|---|---|
| `t-display-xl` | 36 / 600 | Hero, лендинг, success/empty/404/503 |
| `t-heading-l` | 24 / 600 | Заголовки страниц и модалок |
| `t-heading-m` | 20 / 600 | Заголовки карточек, подсекций, настроек |
| `t-title-m` | 16 / 700 | Page title в шапке, особо важные заголовки |
| `t-body-l` | 16 / 400 | Тело модалок, описания, многострочный текст |
| `t-body-s` | 13 / 400 | Helper-текст под инпутами, подсказки |
| `t-label-m` | 13 / 600 | Status-теги, бейджи, account-name |
| `t-caption` | 11 / 400 | Каунтеры, подписи под иконками, даты |
| `t-button-cta` | 16 / 600 | Uppercase-CTA |

`line-height` зашит под роль (tight 1.2 для одно­строчного UI, relaxed 1.5 для параграфов).
В плотном UI (сайдбар, dropdown, dense-таблицы) переопределяем `line-height` на
контейнере, а не вводим новый text-style.

### Плотность интерфейса ЛК = 13px (калибровка по реальному дашборду)

Рабочие экраны кабинета (дашборд, таблицы, формы, навигация, кнопки) — **плотный
13px-first интерфейс**. Дефолтный «body» приложения — **`t-body-s` (13px/400)**, а НЕ
`t-body-l` (16px). 16px (`t-body-l`) — только для модалок, описаний и лендинга.
Измерено на реальном `/dashboard`: body, таблицы, навигация, кнопки — 13px/400; `#555`
(`--text-default`). Не делай рабочие экраны крупнее — будет «не как в сервисе».

Прочие дельты калибровки (реал → канон): шапки таблиц `th` — вес **400** (muted), не 600;
page-title — 16/700; кнопки — 13/400, h40, radius 4px (уже совпадает).

## Шкала размеров `--size-*` (отступы, gap, ширины, иконки)

Всё кратно 2px, имя = значение в px. Tier'ы (см. `tokens.js` → `SIZES` для точного `note`):
`fine` 2–12 · `control` 16–24 · `component` 32–48 · `layout` 64–128.
База карточек — `--size-16`, padding модалок — `--size-24`.

## Новые оси токенов (Фаза 4a)

Помимо цвета/типографики/spacing/radius используй эти токены вместо сырых значений
(полные `note` — в `tokens.js`, группы `OPACITY/MOTION/ZINDEX/CONTROL_HEIGHTS/BREAKPOINTS`):

| Ось | Токены | Когда |
|---|---|---|
| Непрозрачность | `--opacity-disabled .4` · `--opacity-muted .5` · `--opacity-hover .7` · `--opacity-faint .3` | вместо сырых `opacity:` |
| Анимация | `--transition-fast/base/slow` (150/200/300ms) + `--ease-standard/linear` + `--duration-spin/skeleton` | вместо `.2s ease` руками |
| Слои | `--z-dropdown/fixed/overlay/modal/popover/toast/tooltip` (+ `--z-sticky/raised/max`) | вместо магических `z-index` |
| Высота контролов | `--control-height-sm/md/lg` (32/40/48) | кнопки/инпуты/селекты |
| Тень оверлея | `--shadow-overlay` | модалки/поповеры |
| Брейкпоинты | `$bp-sm/md/lg/xl/xxl` (SCSS, в `@media`) | адаптив |
| Notice-варианты | `--notice-success-bg/-text`, `--notice-info-bg/-border` | success/info уведомления |
| Ошибка контрола | `--border-error`, `--text-error`, `--shadow-input-focus-danger` | инпут с ошибкой |

## Адаптивность (обязательно для новых страниц)

Все новые страницы/блоки делаем **mobile-first**: базовые стили — мобайл, расширяем вверх.
Брейкпоинты живут в `resources/scss/_responsive.scss` как SCSS (в `@media` нельзя `var()`).

**Как подключать** — scoped-стили в `.vue` пишем на SCSS и подключаем mixin:
```vue
<style lang="scss" scoped>
@use 'responsive' as *;

.toolbar {            /* база = мобайл: вертикально */
  display: flex; flex-direction: column; gap: var(--size-12);
  @include respond-to(md) { flex-direction: row; align-items: center; }
}
</style>
```
`respond-to(sm|md|lg|xl|xxl)` = min-width; `respond-below(...)` = max-width. Сырые px в `@media` НЕ писать.

**Канонические паттерны** (подробно — в `docs/design-system-canon.md`, раздел «Адаптивность»):
- **Page-padding по брейкпоинтам:** мобайл `--size-32` → планшет `--size-40` → десктоп `--size-48`.
- **Тулбар:** на мобайле колонкой/перенос, кнопки могут схлопываться в иконки; от `md` — в строку.
- **Таблица:** на мобайле (`respond-below(md)`) превращается в список карточек — это **уже принятая в проекте конвенция**: `<td data-title="Заголовок">` + `td::before{content:attr(data-title)}` (так сделаны реальные таблицы в 12 файлах). Используй именно атрибут `data-title` (не `data-label`), чтобы совпадать с реальным кодом. Горизонтальный скролл — запасной вариант.
- **Модалка:** на мобайле во весь экран (или bottom-sheet), от `md` — центрированное окно фикс. ширины.
- **Сетки:** `grid` с `repeat(auto-fill, minmax(...))` либо смена числа колонок по `respond-to`.
