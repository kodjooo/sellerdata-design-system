# Масштабирование дизайн-системы Sellerdata с помощью Opus-агентов

> Как быстро покрыть дизайн-системой все основные разделы кабинета, сохранив
> единые токены/компоненты и точное соответствие реальному продукту.

## 1. Резюме и цель

**Цель.** Получить для каждого ключевого раздела кабинета **экран-эталон**, собранный
исключительно из компонентов `Ds*` и токенов `:root`, выверенный по живому продукту
(computed-diff с заполненным аккаунтом). Параллельно — **достроить библиотеку компонентов**
до полного покрытия и держать всё в синхронизации с отдельным репозиторием
`kodjooo/sellerdata-design-system` (источник для Claude Design).

**Что хотим на выходе:**
- `resources/js/Pages/DesignSystem/Screens/<Section>.vue` — экран-эталон на `Ds*`;
- при необходимости — новые/доработанные компоненты `Ds*` (а не локальный код в экране);
- синхронизированный DS-репозиторий (компоненты + `tokens.json` + экраны);
- каталог `/design-system` пополнен страницами новых компонентов.

**Принцип:** «узнаваемо как в продукте, но без случайного разнобоя» — значения берём 1:1
из реала, сырые/legacy → ближайший токен; повторяющиеся паттерны → компоненты, не хардкод.

## 2. Текущий статус (база)

- **Готово:** токены (`:root`), 23 компонента `Ds*`, каталог `/design-system`
  (Foundations + Компоненты + Паттерны), экран **Дэшборд** (десктоп + мобайл,
  выверен по живому аккаунту, с группировкой/сортировкой/диаграммой/таблицей).
- **Метод проверен** на Дашборде (см. §3).
- **Синхронизация:** `scripts/sync-design-system.sh` (ручной запуск).

## 3. Метод покрытия ОДНОГО раздела (проверенный рецепт)

Для каждого раздела агент выполняет один и тот же цикл:

1. **Доступ к реалу.** Залогиниться в заполненный аккаунт (локально: `UserAutoLoginToken`
   через tinker, URL `/login/auto?token=...&redirect=<route>`; заполненный аккаунт — см.
   `docs/` / память проекта). Открыть реальный раздел.
2. **Снять структуру.** Через Playwright/DOM: какие блоки, в каком порядке, какие
   колонки/метрики/фильтры, состояния (пусто/загрузка/ошибка), мобильная раскладка.
   Прочитать реальный Vue-компонент раздела в `resources/js/Pages|Components/*` —
   там точная композиция и пропсы.
3. **Собрать эталон** `Screens/<Section>.vue` из `Ds*` (`DsAppShell` каркас, `DsTable`,
   `DsSummaryCard/Carousel`, `DsChart`, `DsInfoList`, `DsTabs`, `DsProductCell`, и т.д.),
   данные — реальные (computed-diff).
4. **Нет подходящего компонента/паттерна?** Извлечь его из реала в **новый `Ds*`-компонент**
   (+ страница каталога + роут + сайдбар), а не верстать локально в экране.
5. **Выверка.** Сборка `npm run build` зелёная; Playwright computed-diff эталон↔реал по
   ключевым элементам (шрифты/веса/цвета/структура; отступы — допускается нормализация к токену);
   проверить десктоп И мобайл.
6. **Синхронизация.** `scripts/sync-design-system.sh --ds <path> --branch <prod> --push`.

**Гейт приёмки раздела:** билд зелёный; экран рендерится на десктопе и мобайле; значимое
(структура, набор колонок/метрик, веса/цвета) совпадает с реалом; новый код — в `Ds*`, не в экране;
никаких сырых hex/px вне шкалы.

## 5. Масштабирование через Opus-агентов

### 5.1 Модель параллелизации
- **1 раздел = 1 агент.** Разделы независимы по контенту → можно гонять параллельно.
- **Изоляция:** каждый агент — в своём git worktree (`isolation: "worktree"`), чтобы правки
  не конфликтовали в рабочем дереве.
- **Координатор** (главный поток): раздаёт разделы, собирает результаты, выполняет
  СЕРИАЛИЗОВАННО все правки в общих компонентах и финальную синхронизацию.

### 5.2 Контеншн на общих компонентах (важно!)
`Ds*`-компоненты — общий ресурс. Если два агента одновременно правят `DsTable` — конфликт.
Правило:
- Агент раздела **НЕ коммитит** правки общих `Ds*` сам. Если ему нужен новый компонент или
  доработка существующего — он **возвращает спецификацию** (что за компонент, пропсы, поведение,
  откуда извлечено) + черновик файла.
- **Координатор** применяет правки `Ds*` по одному, собирает, проверяет, что не сломаны другие
  страницы, и только потом отдаёт обновлённый компонент следующим.
- Новые НЕзависимые компоненты (никто другой не трогает) агент может создавать сразу.

### 5.3 Роли
- **Section-агент** (N штук): делает §3 по своему разделу, возвращает: файл `Screens/<X>.vue`,
  список нужных правок/новых `Ds*` (спеки + черновики), отчёт computed-diff.
- **Component-агент** (1, опционально): чистит/обобщает компоненты по спекам от section-агентов.
- **Координатор**: интеграция, билд, синхронизация, разрешение контеншена.

### 5.4 Готовый шаблон промпта для section-агента

```
Ты дорабатываешь дизайн-систему Sellerdata. Сделай экран-эталон раздела «<РАЗДЕЛ>»
(реальный роут <ROUTE>) по методу из docs/ds-scaling-with-opus.md §3.

ОБЯЗАТЕЛЬНО:
1. Прочитай docs/design-system.md и docs/ds-ai-style-guide.md — правила и токены.
2. Залогинься в заполненный аккаунт, открой <ROUTE>, сними структуру (DOM + реальный .vue).
3. Собери resources/js/Pages/DesignSystem/Screens/<Section>.vue ТОЛЬКО из Ds*-компонентов
   (resources/js/Components/Ds/*) и токенов var(--*); данные — реальные.
4. Сверь десктоп И мобайл с реалом (computed-diff: шрифты/веса/цвета/структура/набор колонок).
5. Если нужен новый компонент или правка существующего Ds* — НЕ коммить правку общего компонента;
   верни в отчёте: имя, назначение, пропсы, поведение, откуда извлёк, и черновик .vue.

ЗАПРЕЩЕНО: сырые hex; px вне шкалы --size-*; произвольные радиусы; параллельные «свои» стили
вместо токенов/классов t-*; хардкод повторяющихся паттернов в экране.

ВЕРНИ: путь к Screens/<Section>.vue; список правок/новых Ds* (спеки+черновики);
краткий computed-diff отчёт (что совпало, где намеренная нормализация); статус сборки.
```

### 5.5 Оркестрация (как запускать)
- Запусти параллельно section-агентов на приоритетные разделы (§4), каждый с `isolation: "worktree"`.
- Собери их отчёты. Список нужных правок `Ds*` отдай component-агенту/сделай сам — **по одному**,
  с билдом и проверкой остальных страниц после каждой.
- Подключи новые экраны: роут + пункт сайдбара (группа «Экраны») + (если есть новые компоненты)
  страницы каталога.
- Финал: `npm run build`, прогон Playwright по новым экранам, затем
  `scripts/sync-design-system.sh --push`.

## 6. Доставка
Синхронизация в DS-репозиторий — `scripts/sync-design-system.sh` (см. его шапку и README DS-репо).
Запуск ручной, по готовности пачки разделов.

## 7. Риски и как их гасить
- **Конфликты в общих `Ds*`** → сериализация правок через координатора (§5.2).
- **«Реконструкция вместо копии»** → обязательный computed-diff с живым аккаунтом (§3.5).
- **Расползание токенов** → только `var(--*)`/шкала; нет токена → добавить в `_variables.scss` + `tokens.js`.
- **Раздувание стоимости** → агенты возвращают конечный результат (файл+спеки+отчёт), а не «дампы»;
  параллелим независимое, сериализуем общее.
