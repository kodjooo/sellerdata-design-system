# Масштабирование дизайн-системы Sellerdata с помощью Opus-агентов

> Как быстро покрыть дизайн-системой все основные разделы кабинета, сохранив
> единые токены/компоненты и точное соответствие реальному продукту.

## 1. Резюме и цель

**Цель.** Получить для каждого ключевого раздела кабинета **экран-эталон**, собранный
исключительно из компонентов `Ds*` и токенов `:root`, выверенный по живому продукту
(computed-diff с заполненным аккаунтом). Параллельно — **достроить библиотеку компонентов**
до полного покрытия и держать всё в синхронизации с отдельным репозиторием
`kodjooo/sellerdata-design-system` (источник для Claude Design).

**Что хотим на выходе:**
- `resources/js/Pages/DesignSystem/Screens/<Section>.vue` — экран-эталон на `Ds*`;
- при необходимости — новые/доработанные компоненты `Ds*` (а не локальный код в экране);
- синхронизированный DS-репозиторий (компоненты + `tokens.json` + экраны);
- каталог `/design-system` пополнен страницами новых компонентов.

**Принцип:** «узнаваемо как в продукте, но без случайного разнобоя» — значения берём 1:1
из реала, сырые/legacy → ближайший токен; повторяющиеся паттерны → компоненты, не хардкод.

## 2. Текущий статус (база)

- **Готово:** токены (`:root`), 23 компонента `Ds*`, каталог `/design-system`
  (Foundations + Компоненты + Паттерны), экран **Дэшборд** (десктоп + мобайл,
  выверен по живому аккаунту, с группировкой/сортировкой/диаграммой/таблицей).
- **Метод проверен** на Дашборде (см. §3).
- **Синхронизация:** `scripts/sync-design-system.sh` (ручной запуск).

## 2.5 Окружение и доступы (всё необходимое для автономного запуска)

> Этого раздела достаточно, чтобы агент работал без внешних подсказок.

- **Запуск приложения:** `docker compose up -d` (из корня проекта). Базовый URL — `http://localhost:8080`.
- **Сборка фронта:** `npm run build` на хосте (каталог `public/build` смонтирован в контейнер;
  dev-сервера нет — после правок `.vue`/`.scss` нужен `npm run build`).
- **Заполненный аккаунт для сверки:** пользователь **id 2** (`mark.aborchie@gmail.com`),
  магазин **account 4** (Wildberries, транзакции фев–июнь 2026 — данные есть на всех экранах).
  Демо-аккаунт пустой — НЕ использовать.
- **Сделать account 4 активным** (один раз):
  ```bash
  docker compose exec -T app php artisan tinker --execute="
    DB::table('user_accounts')->where('user_id',2)->update(['active'=>0]);
    DB::table('user_accounts')->where('user_id',2)->where('account_id',4)->update(['active'=>1]);"
  ```
- **Автологин (без пароля):** создать токен и открыть URL:
  ```bash
  docker compose exec -T app php artisan tinker --execute="
    echo App\\Modules\\User\\Models\\UserAutoLoginToken::create([
      'user_id'=>2,'token'=>bin2hex(random_bytes(16)),'expires_at'=>now()->addHour()])->token;"
  # → URL: http://localhost:8080/login/auto?token=<TOKEN>&redirect=<route без слэша, напр. products>
  ```
- **Снятие/сверка вида:** Playwright MCP (`browser_navigate`, `browser_evaluate`,
  `browser_resize`, `browser_take_screenshot`). Computed-diff = сравнить `getComputedStyle`
  ключевых элементов эталона и реала; мобайл — `browser_resize` на 390×840.
- **Палитра для сборки:** готовые компоненты — `resources/js/Components/Ds/*` (23 шт.),
  их живой каталог — `/design-system`; правила и токены — `docs/design-system.md`,
  `docs/ds-ai-style-guide.md`, `resources/scss/_variables.scss` (`:root`).
- **Реальные экраны (исходники для снятия структуры):** `resources/js/Pages/*.vue` и
  `resources/js/Components/<Домен>/*` (напр. `Components/Warehouse/*`, `Components/Advertising/*`).
- **Готовые скрины-эталоны (первоисточник состояний):** `docs/reference/screens/*.png` —
  ~107 снимков живого ЛК по всем разделам и обеим площадкам. Конвенция имён:
  `<раздел>__<состояние>@<площадка>.png` (напр. `товары__список__без-себестоимости@wb.png`,
  `реклама__фильтр-статусы@wb.png`, `расходы__модалка-создание@ozon.png`). Покрыты состояния:
  списки, модалки, фильтры, раскрытые строки/детализация, пустые состояния. Отдельная серия
  `_live__<раздел>__десктоп|мобайл*.png` — эталон десктопной и мобильной раскладок
  (напр. `_live__дашборд__десктоп.png`, `_live__дашборд__мобайл.png`, `_live__склад__мобайл-список.png`).
  Это первичный источник для сверки; Playwright добирает то, чего нет в каталоге.

## 3. Метод покрытия ОДНОГО раздела (проверенный рецепт)

Для каждого раздела агент выполняет один и тот же цикл:

1. **Доступ к реалу.** Поднять приложение и залогиниться в заполненный аккаунт по §2.5
   (account 4, автологин-токеном), открыть реальный раздел `<ROUTE>`.
2. **Снять структуру.** Сначала изучить готовые скрины-эталоны раздела в
   `docs/reference/screens/` (обе площадки `@wb`/`@ozon`, все состояния — модалки, фильтры,
   раскрытые строки, пустые состояния, и серию `_live__…` для десктоп/мобайл) — это первичный
   источник состояний и раскладок. Затем через Playwright/DOM добрать то, чего нет в каталоге:
   какие блоки, в каком порядке, какие колонки/метрики/фильтры, состояния (пусто/загрузка/ошибка),
   мобильная раскладка. Прочитать реальный Vue-компонент раздела в `resources/js/Pages|Components/*` —
   там точная композиция и пропсы.
3. **Собрать эталон** `Screens/<Section>.vue` из `Ds*` (`DsAppShell` каркас, `DsTable`,
   `DsSummaryCard/Carousel`, `DsChart`, `DsInfoList`, `DsTabs`, `DsProductCell`, и т.д.),
   данные — реальные (computed-diff).
4. **Нет подходящего компонента/паттерна?** Извлечь его из реала в **новый `Ds*`-компонент**
   (+ страница каталога + роут + сайдбар), а не верстать локально в экране.
5. **Выверка.** Сборка `npm run build` зелёная; Playwright computed-diff эталон↔реал по
   ключевым элементам (шрифты/веса/цвета/структура; отступы — допускается нормализация к токену);
   проверить десктоп И мобайл.
6. **Синхронизация.** `scripts/sync-design-system.sh --ds <path> --branch <prod> --push`.

**Гейт приёмки раздела:** билд зелёный; экран рендерится на десктопе и мобайле; значимое
(структура, набор колонок/метрик, веса/цвета) совпадает с реалом; новый код — в `Ds*`, не в экране;
никаких сырых hex/px вне шкалы.

## 5. Масштабирование через Opus-агентов

### 5.1 Модель параллелизации
- **1 раздел = 1 агент.** Разделы независимы по контенту → можно гонять параллельно.
- **Изоляция:** каждый агент — в своём git worktree (`isolation: "worktree"`), чтобы правки
  не конфликтовали в рабочем дереве.
- **Координатор** (главный поток): раздаёт разделы, собирает результаты, выполняет
  СЕРИАЛИЗОВАННО все правки в общих компонентах и финальную синхронизацию.

### 5.2 Контеншн на общих компонентах (важно!)
`Ds*`-компоненты — общий ресурс. Если два агента одновременно правят `DsTable` — конфликт.
Правило:
- Агент раздела **НЕ коммитит** правки общих `Ds*` сам. Если ему нужен новый компонент или
  доработка существующего — он **возвращает спецификацию** (что за компонент, пропсы, поведение,
  откуда извлечено) + черновик файла.
- **Координатор** применяет правки `Ds*` по одному, собирает, проверяет, что не сломаны другие
  страницы, и только потом отдаёт обновлённый компонент следующим.
- Новые НЕзависимые компоненты (никто другой не трогает) агент может создавать сразу.

### 5.3 Роли
- **Section-агент** (N штук): делает §3 по своему разделу, возвращает: файл `Screens/<X>.vue`,
  список нужных правок/новых `Ds*` (спеки + черновики), отчёт computed-diff.
- **Component-агент** (1, опционально): чистит/обобщает компоненты по спекам от section-агентов.
- **Координатор**: интеграция, билд, синхронизация, разрешение контеншена.

### 5.4 Готовый шаблон промпта для section-агента

```
Ты дорабатываешь дизайн-систему Sellerdata. Сделай экран-эталон раздела «<РАЗДЕЛ>»
(реальный роут <ROUTE>) по методу из docs/ds-scaling-with-opus.md §3.

ОБЯЗАТЕЛЬНО:
1. Прочитай docs/design-system.md и docs/ds-ai-style-guide.md — правила и токены.
2. Изучи готовые скрины-эталоны раздела в docs/reference/screens/ (обе площадки @wb/@ozon,
   все состояния + серия _live__… для десктоп/мобайл) — это первичный источник состояний и
   раскладок. Затем залогинься в заполненный аккаунт, открой <ROUTE>, добери структуру через
   Playwright (DOM + реальный .vue) — то, чего нет в каталоге.
3. Собери resources/js/Pages/DesignSystem/Screens/<Section>.vue ТОЛЬКО из Ds*-компонентов
   (resources/js/Components/Ds/*) и токенов var(--*); данные — реальные.
4. Сверь десктоп И мобайл с реалом (computed-diff: шрифты/веса/цвета/структура/набор колонок).
5. Если нужен новый компонент или правка существующего Ds* — НЕ коммить правку общего компонента;
   верни в отчёте всё для каталога/доков (координатор заведёт по чек-листу §5.5 шаг 4):
   имя, назначение, ПРОПСЫ (тип/дефолт/описание), СОБЫТИЯ, СЛОТЫ, перечень состояний, откуда извлёк,
   и черновик .vue целиком.

ЗАПРЕЩЕНО: сырые hex; px вне шкалы --size-*; произвольные радиусы; параллельные «свои» стили
вместо токенов/классов t-*; хардкод повторяющихся паттернов в экране.

ВЕРНИ: путь к Screens/<Section>.vue; список правок/новых Ds* (спеки+черновики);
краткий computed-diff отчёт (что совпало, где намеренная нормализация); статус сборки.
```

### 5.5 Оркестрация — конкретные шаги для ведущего Opus-агента

Ведущий агент выполняет это сам, без подсказок:

0. **Подготовка среды:** `docker compose up -d`; сделать account 4 активным и проверить
   автологин (см. §2.5); `npm run build` — убедиться, что база собирается.
1. **Фан-аут.** На приоритетные разделы (§4) запустить **параллельно** section-агентов
   инструментом `Agent` с `subagent_type: "sd-ui"` (или `general-purpose`) и
   **`isolation: "worktree"`** (изоляция правок). Каждому дать промпт из §5.4, подставив
   `<РАЗДЕЛ>`/`<ROUTE>`. Несколько `Agent`-вызовов — в одном сообщении (идут параллельно).
   Для строгого пайплайна можно использовать инструмент `Workflow` (фан-аут → сбор → интеграция).
2. **Сбор.** Принять от каждого: `Screens/<Section>.vue`, список спек/черновиков `Ds*`,
   отчёт computed-diff. Перенести файлы экранов из worktree в основную ветку.
3. **Интеграция общих `Ds*` — СТРОГО последовательно** (см. §5.2): применять правки/новые
   компоненты по одному, после каждого — `npm run build` и быстрый Playwright-смоук
   затронутых страниц каталога, чтобы не сломать другие экраны.
4. **Подключение экранов:** роут в `routes/web.php` (`designSystem.screen<X>`) + пункт сайдбара
   в `DocsSidebar.vue` (группа «Паттерны»/«Экраны»).
   **Новый/изменённый общий `Ds*` НЕ считается готовым, пока не проведён по полному чек-листу
   (иначе компонент «невидим» в каталоге и доках):**
   - [ ] файл `resources/js/Components/Ds/Ds<X>.vue` (только токены/классы `t-*`, без сырых hex/px);
   - [ ] страница каталога `Pages/DesignSystem/Components/<X>.vue` (`DocsLayout` + живой `DocExample` + `PropsTable`; события — отдельной таблицей);
   - [ ] роут `designSystem.<x>` в `routes/web.php`;
   - [ ] пункт в `DocsSidebar.vue` (группа «КОМПОНЕНТЫ»);
   - [ ] строка в реестре `docs/design-system-components.md` (таблица props/состояния/файл + строка slots/emits);
   - [ ] **изменение существующего компонента** (новый проп/слот/событие) — дописать в его страницу каталога (демо + строка в `PropsTable`) и в `docs/design-system-components.md`;
   - [ ] новый токен (если вводился) — в `_variables.scss` (`:root`) и `tokens.js`; иначе явно отметить «новых токенов не вводилось».
5. **Приёмка:** `npm run build` зелёный; Playwright по каждому новому экрану на десктопе (1440)
   и мобайле (390) — структура/колонки/веса/цвета совпадают с реалом (§3.5).
6. **Доставка:** закоммитить в ветку приложения и запушить, затем
   `scripts/sync-design-system.sh --ds <путь-к-клону> --branch <ветка> --push`.

Параллелизм безопасен для контента (разные экраны/данные), опасен для общих `Ds*` —
поэтому шаг 1 параллельный, шаг 3 строго серийный.

## 6. Доставка
Синхронизация в DS-репозиторий — `scripts/sync-design-system.sh` (см. его шапку и README DS-репо).
Запуск ручной, по готовности пачки разделов.

## 7. Риски и как их гасить
- **Конфликты в общих `Ds*`** → сериализация правок через координатора (§5.2).
- **«Реконструкция вместо копии»** → сверка с готовыми скринами `docs/reference/screens/`
  (все состояния + десктоп/мобайл) + обязательный computed-diff с живым аккаунтом (§3.5).
- **Расползание токенов** → только `var(--*)`/шкала; нет токена → добавить в `_variables.scss` + `tokens.js`.
- **Раздувание стоимости** → агенты возвращают конечный результат (файл+спеки+отчёт), а не «дампы»;
  параллелим независимое, сериализуем общее.
